## 🎯 Project Overview

Guardian Pro demonstrates the complete attack-defence lifecycle of a brute-force intrusion attempt against a password-protected file vault:

| Role | Module | What it does |
|------|--------|-------------|
| **Defender** | `login.py` | Hashes passwords, tracks failures, triggers self-destruct |
| **Attacker** | `brute.py` | Systematically tries all 1,000 combinations |
| **Web UI** | `server.py` | Real-time browser dashboard with live simulation |
| **Demo** | `run_demo.py` | Guided walkthrough — one command covers everything |
| **Launcher** | `guardian_pro.py` | Single entry point for all modes |

### Learning Objectives

- Understand **why** brute-force attacks succeed on weak password spaces
- See **how** defenders use rate-limiting, lockout, and destruction to counter them
- Learn **cryptographic hashing** — why passwords are never stored in plaintext
- Observe **secure multi-pass file wiping** (DoD 5220.22-M inspired)
- Analyse a **forensic audit trail** after an incident

## 🔑 Key Concepts Demonstrated

### 1. Cryptographic Hashing
```python
import hashlib
# Password "742" becomes a 64-character hex digest
hash = hashlib.sha256("742".encode()).hexdigest()
# → "4e07408562bedb8b60ce05c1decfe3ad16b72230967de01f640b7e4729b49fce"
```
- **One-way function** — the hash cannot be reversed to recover "742"
- **Deterministic** — the same input always produces the same hash
- **Avalanche effect** — changing one character changes ~50% of the output

### 2. Rate Limiting
```
Attempt 1 → FAIL → wait 2s
Attempt 2 → FAIL → wait 2s
Attempt 3 → FAIL → SELF-DESTRUCT
```
Each 2-second delay multiplies attack time by the delay factor. At 2s/attempt, cracking a 1,000-word space takes up to **33 minutes** instead of milliseconds.

### 3. Intrusion Detection
The system tracks a `failed_attempts` counter. Exceeding `MAX_ATTEMPTS` (default: 3) is the detection threshold — no IP logs or network needed.

### 4. Secure Multi-Pass File Deletion
```
Pass 1–3 : Overwrite with cryptographically random bytes  (os.urandom)
Pass 4   : Overwrite with null bytes  (b"\x00" × file_size)
Pass 5   : os.remove() — unlink inode
```
Simple deletion only removes the filesystem pointer. The data remains on disk until overwritten. Multi-pass wiping ensures recovery tools (e.g., `foremost`, `photorec`) cannot retrieve the original content.

### 5. Audit Logging
Every event is timestamped and appended to `access_log.txt`:
```
[2025-04-13 10:30:15] AUTH_FAILURE — Attempted: '000' | Strike 1/3
[2025-04-13 10:30:17] AUTH_FAILURE — Attempted: '001' | Strike 2/3
[2025-04-13 10:30:19] AUTH_FAILURE — Attempted: '002' | Strike 3/3
[2025-04-13 10:30:20] INTRUSION_DETECTED — Self-destruct initiated.
[2025-04-13 10:30:22] SECURITY_EVENT — 'protected.txt' destroyed.
```

## 🛡️ How the Defence Works

```
User/Attacker enters password
         │
         ▼
   System locked? ──► YES ──► "Access denied"  [END]
         │ NO
         ▼
   File missing? ──► YES ──► Emergency lock    [END]
         │ NO
         ▼
   Hash(input) == stored_hash?
         │
   YES ──┼──► Reset counter, grant access      [SUCCESS]
         │
   NO ───┼──► failed_attempts += 1
         │
         ├── failed_attempts < MAX_ATTEMPTS
         │       └─► Apply rate-limit delay, allow retry
         │
         └── failed_attempts == MAX_ATTEMPTS
                 └─► SELF-DESTRUCT SEQUENCE
                         ├─ 3× random overwrite
                         ├─ 1× null overwrite
                         ├─ os.remove()
                         └─ system_locked = True
```

## ⚡ How the Attack Works

```
brute_force_attack()
       │
       ▼
Initialize target system
       │
       ▼
for password in ["000", "001", …, "999"]:
       │
       ├── Check: system still unlocked?  NO → ABORT (system defended)
       ├── Check: file still exists?      NO → ABORT (file destroyed)
       │
       └── login.check_password(password)
                 │
            True ──► VICTORY: display vault contents
                 │
           False ──► Continue to next candidate
                             │
                       after ATTACK_DELAY seconds
```

**Why the attacker always loses with default settings:**

The password "742" is at index 743. But the system only allows 3 attempts — the attacker is detected and the file is destroyed at attempt #3, long before reaching #743.

**To make the attacker win:** change `CORRECT_PASSWORD = "000"` in `login.py` so it's found on the very first try.

## 🌐 Web Dashboard Guide

1. **Configuration tab**  
   Enter the absolute path to any file on your system, a passphrase, and choose how many failures trigger lockout. Toggle **Simulation** (safe, resets counter) vs **Live** (irreversible file destruction).

2. **Secure Vault tab**  
   The strike meter shows how many failures have occurred. Enter the passphrase to decrypt and display the vault. Wrong password? Strikes accumulate — hit the limit and the overlay appears.

3. **Attack Simulator tab**  
   Click *Execute Brute Force* to watch a live progress bar, attempt counter, and speed meter as the system tries every combination from 000→999 in real time.

4. **Audit Logs tab**  
   Full event stream with colour-coded entries. Refreshes automatically every 3 seconds.

## ⚙️ Customisation

### Change the password
```python
# login.py — line 34
CORRECT_PASSWORD: str = "742"   # Change to any value
```

### Widen the search space
```python
# brute.py — lines 33–34
PASSWORD_MIN: int = 0
PASSWORD_MAX: int = 9999   # 4-digit search (10,000 candidates)
```

### Increase the attempt limit
```python
# login.py — line 42
MAX_ATTEMPTS: int = 5   # Allow 5 failures before lockout
```

### Remove the rate-limit delay
```python
# login.py — line 43
DELAY_SECONDS: int = 0   # No delay (educational: shows attack speed)

# brute.py — line 37
ATTACK_DELAY: float = 0   # No attacker delay either
```

```python
# login.py — line 39
PROTECTED_FILE: str = "/path/to/your/secret.txt"
```

## 📋 Understanding the Audit Log

```
[2025-04-13 10:30:15] SYSTEM_ARMED | Mode=SIM | Target='protected.txt' | MaxAttempts=3
[2025-04-13 10:30:16] AUTH_FAILURE — Attempted: '000' | Strike 1/3
[2025-04-13 10:30:18] AUTH_FAILURE — Attempted: '001' | Strike 2/3
[2025-04-13 10:30:20] AUTH_FAILURE — Attempted: '002' | Strike 3/3
[2025-04-13 10:30:20] INTRUSION_DETECTED — Self-destruct sequence initiated.
[2025-04-13 10:30:22] SECURITY_EVENT — 'protected.txt' destroyed (intrusion response).
```

| Event keyword | Meaning |
|---------------|---------|
| `SYSTEM_ARMED` | Configuration saved, system ready |
| `AUTH_SUCCESS` | Correct password entered |
| `AUTH_FAILURE` | Wrong password; strike count shown |
| `INTRUSION_DETECTED` | Threshold exceeded; wipe triggered |
| `SECURITY_EVENT` | File actually deleted from disk |
| `SYSTEM_RESET` | State cleared (fresh start) |
| `OS_ERROR` | Deletion encountered a filesystem error |

**To reset for a fresh run:**
```bash
rm -f access_log.txt protected.txt guardian_state.json
```

## 🔬 Security Analysis

### Vulnerabilities in This System (By Design)

| Vulnerability | Why It Exists | Real-World Fix |
|---------------|---------------|----------------|
| Tiny password space (0–999) | Keeps the demo fast | Use 12+ character alphanumeric passwords |
| SHA-256 without salt | Simplicity for teaching | Use bcrypt / Argon2 with per-user salts |
| No IP-based rate limiting | Single-machine demo | Nginx / firewall rules; fail2ban |
| Fixed delay (not exponential) | Easy to predict | Exponential back-off + jitter |
| No CAPTCHA | Automated scripts run freely | hCaptcha, reCAPTCHA, proof-of-work |
| No MFA | Single factor only | TOTP / WebAuthn as second factor |

### Defences That Work Well

| Defence | Implementation | Effectiveness |
|---------|---------------|---------------|
| SHA-256 hashing | `hashlib.sha256` | Prevents plaintext leakage |
| Attempt counter | `failed_attempts` global | Limits attack window to 3 tries |
| Rate-limit delay | `time.sleep(2)` | Multiplies total attack time ×2000 |
| Self-destruct | Multi-pass wipe + `os.remove` | Data unrecoverable after threshold |
| Audit logging | Timestamped append | Full forensic trail survives wipe |

## 🐛 Troubleshooting

**`ModuleNotFoundError: No module named 'login'`**  
Both files must be in the same directory.
```bash
ls *.py   # Should list login.py and brute.py
```

**`ModuleNotFoundError: No module named 'uvicorn'`**  
```bash
pip install uvicorn starlette
```

**Attack stops immediately with "file not found"**  
The protected file was already deleted from a previous run. Reset:
```bash
rm -f protected.txt access_log.txt
python3 brute.py
```

**Want the attacker to WIN for demonstration?**  
Change the password to the first value in the search range:
```python
# login.py
CORRECT_PASSWORD: str = "000"
```

**Output scrolls too fast?**  
Increase the attack delay:
```python
# brute.py
ATTACK_DELAY: float = 0.5   # Half-second between attempts
```
